The Three Hard Problems / 三个核心难题

01

Ground Truth Problem / 真值问题

Before you can benchmark, you need to define "correct". For corner cases, the algorithm already failed — so what IS the correct answer? Who labels it? How do you maintain it over time?

做 Benchmark 的前提是知道"正确答案"。但 corner case 本身就是算法出错的场景 — 那正确答案谁来定义？怎么标注？怎么维护？

Hardest — solve first / 最难 — 优先解决
02

Standardization Problem / 标准化问题

Raw InfluxDB data (tar.gz) is not directly consumable by algorithms. You need a transformation layer that converts raw sensor data into the exact input format each algorithm service expects.

InfluxDB 原始时序数据（tar.gz）不能直接喂给算法。需要一个转换层，把传感器数据转成算法服务期望的输入格式。

Medium — engineering work / 中等 — 工程量
03

Orchestration Problem / 编排问题

Running benchmarks means coordinating: data prep, service calls, result collection, comparison, and reporting. Across multiple algorithm versions, multiple test cases, potentially in parallel.

跑 Benchmark 需要协调：数据准备、服务调用、结果收集、比对、报告。跨多个算法版本、多个用例，可能并行执行。

Tractable — you've done this / 可控 — 你有经验

Key Questions Before You Build / 动手前的关键问题

Q1: Who defines ground truth? / 谁定义真值？

Options: Human annotation (domain experts label correct output), Simulation replay (use simulator output as reference), Golden version (freeze a "known good" algorithm as baseline), or Hybrid (auto-baseline + human review for edge cases).

Q2: What gets compared? / 比较什么？

Algorithm output could be: trajectories, control signals, object detections, planning decisions. Each needs different comparison metrics (L2 distance, IoU, decision match rate, etc.).

Q3: Where does Benchmark run? / 在哪跑？

Your diagram shows Dagster/Java. Is this existing infra? Airflow is at the site, Benchmark is at HQ — different environments. Do you reuse Airflow or build something separate?

Q4: What's the MVP? / 最小闭环是什么？

Simplest useful benchmark: take 10 corner cases → run through algorithm → human reviews output → marks pass/fail. No automation needed at first. Validate the workflow before building infra.

Suggested Roadmap / 建议路线图

Phase 0
1-2 weeks

Close the Loop Manually / 手动走通全流程

Don't build anything yet. Take 5-10 packaged corner cases, manually run through the algorithm, manually compare results. The goal: understand the actual workflow before automating it. Document what inputs the algorithm needs, what outputs it produces, and how you'd judge success.

先别写代码。拿 5-10 个已打包的 corner case，手动跑一遍算法，手动比对结果。目标：在自动化之前，搞清楚真实工作流。记录算法需要什么输入、产出什么输出、怎么判断成功。

Phase 1
2-3 weeks

Ground Truth + Catalog / 真值管理 + 用例目录

Extend auto_test_case_catalog with ground_truth_json. Build a simple annotation tool or workflow (even a spreadsheet works). Evolve process_status: PACKAGED → LABELED → BENCHMARKED. This is the foundation everything else builds on.

Phase 2
3-4 weeks

Data Transformer + Runner MVP / 数据转换器 + 执行器 MVP

Build the "standardized materials" converter (tar.gz → algorithm input format). Build a simple benchmark runner that calls the algorithm API, captures output, and stores results. Start with a single algorithm, a single comparison metric.

Phase 3
2-3 weeks

Comparison Engine + Reports / 比对引擎 + 报告

Automated ground truth comparison with configurable metrics. Version-over-version regression reports. Dashboard showing algorithm performance trends. This is where the real business value starts flowing.

Phase 4
future

Training Pipeline + Active Learning / 训练管线 + 主动学习

Curate corner cases into training datasets. Feed back into model training. Active learning: model identifies its own weak spots, requests more data from the field. This closes the full loop from the original architecture vision.
